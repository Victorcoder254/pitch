§ 02

Why this infrastructure has value

The components below are individually non-trivial. Each required real iteration to produce correctly. A buyer acquiring this codebase skips that iteration entirely.

Component What was solved
WebRTC Signalling Full offer / answer / ICE candidate relay through Django Channels. P2P video with ICE watchdog timeout and auto-rematch. No media server — no per-minute streaming cost.
Redis Matchmaking Vibe-scoped pools with atomic SPOP. Supports casual chat, per-game vibes, and is trivially extended to any pairing model: dating, language exchange, interest groups.
Presence System Live online user count pushed via a dedicated /ws/presence/ consumer to any subscribed page. Zero polling. Fully decoupled from match logic.
Game Plugin Framework Four games shipped. Adding a fifth requires: one GameType row, one JS engine file implementing init(), handleRemoteMove(), and destroy(). Roles, scoring, and sync are already built.
Authentication django-allauth with Google OAuth and email/password. Custom user model with terms and age gate before access to /connect/. Extensible to any social provider allauth supports.
Data Model MatchSession, GameResult, GameType, CustomUser — fully documented. GameResult rows are write-once. Leaderboards and profiles are one aggregation query away.
Production Deployment Nginx reverse proxy, Daphne ASGI, systemd socket activation, and deployment scripts. Not a tutorial setup — this is the configuration required to actually run a Django Channels application in production.
Moderation Hook report_peer event writes a record with timestamp and reporter ID. Filterable in Django admin. A moderation queue view is one view away — the hook is already in place.

System architecture

The server handles matchmaking, signalling, presence, and game synchronisation. Video and audio traffic flows peer-to-peer once ICE negotiation completes — the server exits the media path entirely.

Sessions, users, game results
WebSocket endpoints
/ws/match/

MatchmakingConsumer

Handles matchmaking, WebRTC signalling, in-session chat, game move relay, peer reporting, and session lifecycle. This is the primary application consumer.

/ws/presence/

PresenceConsumer

Read-only. Pushes a live online_count broadcast to any subscribed page without polling. Decoupled from match logic.

Vibe system

Intent-based routing

Users declare intent before matching. casual routes to general video chat. board_game:<code> routes to a game-specific pool and auto-launches the engine on match. Adding a new vibe requires one Redis pool and one routing entry.

§ 09

Expansion roadmap

These are the natural next development phases. In most cases, the data structures and hooks already exist — they are a matter of building the UI layer on top of what's there.

R-01
Leaderboards and user profiles

GameResult rows are already written on session close. An aggregation query and a profile view is all that is required.

R-02
Chess engine

Referenced in the existing codebase. The plugin system is ready. Requires a JS engine implementing the three-method interface.

R-03
TURN server integration

Adds relay support for symmetric NAT users, estimated at 15–20% of connections. Coturn is the standard open-source solution. Configuration is a one-time infrastructure task.

R-04
Group and party sessions

Requires a room/lobby model and a multi-user consumer. The channel layer infrastructure supports it — it is a new consumer, not an architecture change.

R-05
Advanced moderation tools

The reported=True flag and timestamp already exist on MatchSession. An admin queue view is one Django view and a template.

R-06
Mobile applications

The existing frontend is responsive. Progressive web app enhancement is available immediately. A React Native client would consume the same WebSocket protocol documented in the technical reference.

R-07
Streaks, achievements, and engagement systems

Build on the existing GameResult scoring model. The data is already there — the surface layer is not.
